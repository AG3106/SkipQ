1  Context  Design 
1.1  Context  Model  
The  context  model  illustrates  how  the  system  interacts  with  external  entities  and  services  in  
its
 
operating
 
environment,
 
along
 
with
 
the
 
relationships
 
between
 
them.
 
It
 
deﬁnes
 
the
 
system
 
boundaries
 
and
 
highlights
 
all
 
external
 
dependencies
 
required
 
for
 
proper
 
functioning.
 
This
 
model
 
ensures
 
that
 
interactions
 
with
 
users,
 
administrative
 
roles,
 
and
 
third-party
 
services
 
are
 
clearly
 
identiﬁed
 
and
 
taken
 
care
 
of
 
during
 
system
 
design.
  A  brief  explanation  of  the  context  model  is  as  follows:  ●  User/Customer:   Interacts  with  the  system  through  the  website  to  browse  menu  items,  place  orders,  
make
 
payments,
 
and
 
track
 
order
 
status.
  ●  Canteen  Manager:   Uses  the  website  to  manage  incoming  orders,  update  menu  items,  and  modify  prices  
in
 
the
 
database.
  ●  Admin:   Monitors  and  oversees  the  overall  functioning  of  the  platform,  ensuring  smooth  
operation,
 
resolving
 
system
 
issues,
 
and
 
enforcing
 
platform
 
policies.
  ●  Database:  
 
Stores
 
all
 
system
 
data,
 
including
 
user
 
credentials,
 
order
 
details,
 
transaction
 
records,
 
available
 
items
 
and
 
system
 
logs.
 
 ●  Mail  Server:  
 
Facilitates
 
system
 
communication
 
by
 
sending
 
One-Time
 
Passwords
 
(OTPs),
 
order
 
conﬁrmations,
 
and
 
supporting
 
password
 
recovery
 
through
 
email
 
veriﬁcation.
 
 ●  Payment  Gateway:  
 
Integrates
 
secure
 
payment
 
functionality,
 
enabling
 
online
 
ﬁnancial
 
transactions
 
and
 
conﬁrming
 
payment
 
status
 
between
 
users
 
and
 
the
 
system.
 
 ●  Authentication  Providers:  
 
Manages
 
user
 
authentication,
 
ensuring
 
secure
 
login
 
and
 
OTP-based
 
veriﬁcation.
 
 ●  Web  Server:  
 
Acts
 
as
 
the
 
central
 
system
 
component
 
that
 
processes
 
user
 
requests,
 
enforces
 
business
 
logic,
 
and
 
coordinates
 
communication
 
between
 
external
 
services
 
and
 
the
 
database.

2  Architecture  Design 
 2.1  Purpose   This  section  describes  the  high-level  architectural  design  of  the  SkipQ  system.  It  
outlines
 
the
 
major
 
system
 
components,
 
their
 
responsibilities,
 
and
 
the
 
interactions
 
between
 
them.
 
The
 
objective
 
of
 
this
 
section
 
is
 
to
 
provide
 
a
 
clear
 
structural
 
view
 
of
 
the
 
system
 
to
 
support
 
implementation,
 
maintenance,
 
and
 
future
 
enhancements.
  2.2  Architectural  Overview   SkipQ  is  designed  using  a  layered  client–server  architecture  to  support  multiple  user  
roles,
 
including
 
customers,
 
canteen
 
managers,
 
and
 
administrators.
 
The
 
system
 
is
 
implemented
 
as
 
a
 
centralized
 
web-based
 
platform
 
accessible
 
through
 
modern
 
web
 
browsers
 
on
 
smartphones,
 
tablets,
 
and
 
desktop
 
devices.
  The  architecture  separates  user  interaction,  application  control,  business  logic,  and  data  
storage
 
into
 
distinct
 
layers
.
 
This
 
separation
 
improves
 
modularity,
 
scalability,
 
and
 
maintainability
 
while
 
enabling
 
secure
 
integration
 
with
 
external
 
services
 
such
 
as
 
payment
 
gateways
 
and
 
future
 
machine
 
learning
 
components.
  2.3  Architectural  Style   The  system  follows  a  Layered  Architecture  with  modular  backend  services .  Each  layer  
performs
 
a
 
specific
 
role
 
and
 
communicates
 
with
 
adjacent
 
layers
 
through
 
well-defined
 
interfaces.
  The  architectural  design  supports:   ●  Loose  coupling  between  frontend  and  backend  components   ●  Secure  communication  using  HTTPS   ●  Scalability  to  support  concurrent  users  during  peak  hours   ●  Future  extensibility  for  analytics  and  machine  learning  features

 2.5  Component-Level  Design   2.5.1  Presentation  Layer   Description:   The  Presentation  Layer  provides  the  user-facing  interfaces  for  all  system  users.   Components:   ●  Customer  Web  Interface   ●  Canteen  Manager  Web  Interface   ●  Admin  Dashboard   
 

Software  Design  Document  for  <Group  9>   Page  7  
Responsibilities:   ●  User  authentication  and  session  handling   ●  Displaying  menus,  orders,  queues,  and  statistics   ●  Collecting  user  inputs  such  as  orders,  payments,  and  reviews   ●  Displaying  real-time  updates  on  order  status   Interfaces:   Communicates  with  the  backend  server  using  REST  APIs  over  HTTPS.   2.5.2  Application  Layer   Description:   The  Application  Layer  acts  as  the  intermediary  between  the  Presentation  Layer  and  the  
Business
 
Logic
 
Layer.
 
It
 
is
 
responsible
 
for
 
handling
 
client
 
requests
 
and
 
coordinating
 
system
 
operations
 
in
 
a
 
secure
 
and
 
controlled
 
manner,
 
and
 
is
 
also
 
responsible
 
for
 
authentication
,
 
authorisation
,
 
request
 
validation
,
 
and
 
rate-limiting
.
  Responsibilities:   ●  Handling  incoming  API  requests  from  frontend  interfaces   ●  Performing  request  validation  and  input  sanitisation   ●  Managing  authentication  and  authorisation   ●  Routing  requests  to  appropriate  business  logic  modules   ●  Implementing  request  logging  and  rate-limiting  mechanisms  to  ensure  system  
reliability
 
under
 
high
 
load
  Interfaces:   ●  REST  APIs  for  frontend  communication.   ●  Database  access  interfaces.   ●  External  service  APIs.     
 
Software  Design  Document  for  <Group  9>   Page  8  
2.5.3  Business  Logic  Layer   Description:   The  Business  Logic  Layer  implements  the  core  functional  behaviour  of  the  SkipQ  
system.
  Submodules  include:   a)  Order  Management  Module   ●  Handles  order  placement,  acceptance,  rejection,  and  completion   ●  Manages  order  queues  and  status  updates   b)  Wallet  and  Payment  Module   ●  Manages  SkipQ  Wallet  balance   ●  Handles  wallet  transactions  and  payment  gateway  integration   ●  Updates  transaction  history  and  monthly  expenditure  statistics   c)  Menu  and  Canteen  Management  Module   ●  Manages  menus,  prices,  availability,  and  canteen  schedules   d)  Review  and  Favourites  Module   ●  Handles  ratings,  reviews,  and  favourites  management   ●  Ensures  reviews  are  allowed  only  for  completed  orders   e)  Admin  Management  Module   ●  Manages  canteen  onboarding  and  verification   ●  Handles  user  moderation  and  global  analytics   2.5.4  Data  Layer   Description:  The  Data  Laye r  provides  persistent  storage  for  all  system  data.     
 
Software  Design  Document  for  <Group  9>   Page  9  
Responsibilities:   ●  Storing  user,  canteen,  order,  payment,  and  review  data.   ●  Maintaining  transaction  records.   ●  Supporting  analytics  and  future  machine  learning  models.   Storage  Technologies:   PostgreSQL  or  MongoDB  database  management  systems.   2.5.5  External  Systems   Payment  Gateway:   External  payment  services  (e.g.,  Razorpay/UPI)  are  integrated  for  secure  transaction  
processing
 
using
 
APIs.
 
All
 
payment-related
 
communication
 
is
 
handled
 
exclusively
 
through
 
the
 
Wallet
 
and
 
Payment
 
Module
 
to
 
ensure
 
transactional
 
consistency
 
and
 
security.
  Isolation  Constraint:   All  third-party  integrations  are  isolated  from  the  core  business  logic  of  the  system.  Any  
failure
 
or
 
timeout
 
in
 
external
 
services
 
does
 
not
 
directly
 
affect
 
internal
 
order
 
management
 
or
 
data
 
consistency.
  Future  Machine  Learning  Services:   The  architecture  supports  future  integration  of  recommendation  systems  and  wait-time  
prediction
 
models
 
as
 
independent
 
services.
 
These
 
services
 
will
 
consume
 
historical
 
data
 
from
 
the
 
Data
 
Layer
 
without
 
interfering
 
with
 
real-time
 
system
 
operations.
  2.6  Data  Flow  Description   ●  A  user  initiates  an  action  through  the  web  interface.  ●  The  request  is  sent  to  the  backend  server  via  HTTPS.  ●  The  backend  validates  the  request  and  invokes  appropriate  business  logic  modules.  ●  If  required,  the  backend  communicates  with  external  payment  services.  ●  Upon  successful  processing,  data  is  updated  in  the  database.  ●  Real-time  status  updates  are  sent  to  relevant  users.  ●  Transaction  and  order  data  contribute  to  statistics  and  analytics.    2.7  Architectural  Constraints   The  architecture  is  designed  to  comply  with  the  following  constraints:  
 
Software  Design  Document  for  <Group  9>   Page  10  
 ●  Support  for  a  minimum  of  500  concurrent  users   ●  Secure  handling  of  authentication  and  financial  data   ●  Modular  design  to  allow  transition  from  fake  wallet  to  real  payment  gateway   ●  Compatibility  with  modern  web  browsers  only   ●  Support  for  future  feature  expansion  without  major  redesign   2.8  Design  Rationale   A  layered  architecture  was  selected  to  ensure  separation  of  concerns ,  simplify  
maintenance
 
and
 
future
 
scalability
.
