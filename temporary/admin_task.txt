# Administration Module: Current State and Remaining Tasks

## Overview
The `administration` module in SkipQ is responsible for the overall monitoring, moderation, and control of the platform. It enforces policies, manages user access, and oversees canteen onboarding. The module functions as the global control plane for the campus canteen food ordering platform.

### Role in the Project
As defined in the SDD and CONTEXT.md, the Admin acts to:
- Oversee canteen registrations (Approve/Reject requests).
- Manage user violations (Suspend/Unsuspend users).
- Provide system-wide analytics (users, orders, revenue, canteens).
- Moderate content (delete inappropriate reviews or reject non-compliant canteens).
- Communicate with the user base via broadcasts.
- Maintain an activity log of administrative actions for auditing purposes.

---

## Current Implementation Status
The `Backend/apps/administration` folder currently implements the following endpoints in `views.py` (and mapped in `urls.py`):

1. **Canteen Management**
   - `GET /canteen-requests/`: view pending canteen requests.
   - `POST /canteen-requests/<id>/approve/`: approve a canteen.
   - `POST /canteen-requests/<id>/reject/`: reject a canteen.
2. **User Management**
   - `GET /users/`: list all users.
   - `POST /users/<id>/suspend/`: suspend a user.
   - `POST /users/<id>/unsuspend/`: unsuspend a user.
3. **Analytics & Logging**
   - `GET /analytics/`: view global statistics (revenue, total orders, users, canteens).
   - `GET /activity-log/`: view the admin's action history.
4. **Platform Control**
   - `POST /broadcast/`: send a global message (currently a stub/mock).
   - `POST /moderate/`: delete a `review` or `canteen`.

*Note: The `admin.py` is minimal, `models.py` is virtually empty (Admin models reside in the `users` app), and `tests.py` is empty.*

---

## What is Missing / Remaining Tasks

Based on the SDD requirements, the CONTEXT.md notes, and best practices for the implemented views, the following tasks remain for the `administration` module:

### 1. Actual Broadcast Integration
- **Context**: The `broadcast_notification` endpoint currently just logs the message to the console and counts the recipients.
- **Task**: Integrate a real communication service (e.g., Email via Django `send_mail`, WebSocket for real-time in-app alerts, or a Push Notification service) so messages actually reach users.

### 2. Comprehensive Unit Tests
- **Context**: The `tests.py` file is empty (63 bytes) and the context explicitly notes that only bash/curl integration tests exist. 
- **Task**: Write comprehensive Django unit tests or DRF `APITestCase` tests for all admin endpoints. This must include:
  - Role-based access control (ensuring non-admins get 403 Forbidden).
  - Valid and invalid state transitions (e.g., approving an already approved canteen).
  - Verification of activity logging during moderation or suspension actions.

### 3. Pagination for List Endpoints
- **Context**: Endpoints like `list_users` (`User.objects.all()`) and `pending_canteen_requests` return unpaginated querysets.
- **Task**: Implement REST Framework pagination to ensure the system scales efficiently as the number of users and orders grows.

### 4. Advanced Moderation Workflows
- **Context**: Content moderation currently only supports hard deletion (`action="delete"`).
- **Task**: 
  - Allow hiding/flagging content instead of immediately deleting it.
  - Automatically notify the user (customer/manager) when their content (review/canteen) is deleted by an admin, including the reason provided.

### 5. Analytics Optimization
- **Context**: The `global_analytics` endpoint performs multiple synchronous aggregation queries.
- **Task**: Implement database indexing or query caching to prevent this endpoint from causing performance bottlenecks under heavy load.

### 6. Canteen Documents Handling
- **Context**: CONTEXT.md states "File upload handling for canteen documents" is not yet done.
- **Task**: The admin approval process requires verifying documents. Ensure that the admin view can access or securely fetch uploaded canteen verification documents (this intersects with the `canteens` module but is critical for admin approval workflows).
